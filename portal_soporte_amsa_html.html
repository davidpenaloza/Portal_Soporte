Modelo operativo de soporte

Define cómo opera soporte, qué responsabilidades tiene cada rol y cómo se conectan monitoreo, diagnóstico, comunicación y escalamiento.

Modelo AMSA

1. Soporte preventivo

Monitoreo proactivo de pipelines, ingestas, servicios, logs, dashboards y señales de negocio antes de que se materialice un incidente.

2. Soporte reactivo

Atención de incidentes reportados, revisión de evidencia, diagnóstico inicial, comunicación y escalamiento según matriz definida.

3. Mejora continua

Lecciones aprendidas, actualización de runbooks, optimización de alertas, reducción de ruido y fortalecimiento del modelo de observabilidad.

Principios del estándar: continuidad operacional, trazabilidad extremo a extremo, claridad de responsabilidades, documentación viva, monitoreo por capas, escalamiento oportuno y aprendizaje continuo.

Gestión de incidentes

Flujo estándar para detectar, validar, diagnosticar, comunicar, escalar y cerrar incidentes de soporte.

Flujo operativo

Detectar

Alerta, reporte usuario, dashboard o monitoreo proactivo.

Validar

Confirmar alcance, impacto, horario, producto y criticidad.

Diagnosticar

Revisar logs, ejecuciones, dependencias, datos y últimas liberaciones.

Escalar

Derivar a plataforma, desarrollo, negocio o proveedor según matriz.

Cerrar

Documentar causa, solución, evidencia, acciones preventivas y aprendizaje.

Comunicación sugerida: informar estado inicial, impacto conocido, acciones en curso, responsable del seguimiento y próxima actualización estimada. Evitar conclusiones sin evidencia.

Runbooks operativos

Guías paso a paso para diagnosticar incidentes frecuentes, reducir dependencia individual y estandarizar la respuesta del equipo.

Plantillas editables
Runbook 01 · Producto sin datos actualizados
  • Validar si el problema afecta a todos los usuarios o solo a un dashboard específico.
  • Revisar última ejecución correcta de ingesta o pipeline.
  • Comparar frecuencia esperada versus última actualización real.
  • Consultar logs técnicos en Log Analytics Workspace.
  • Identificar si hay dependencia externa: fuente, storage, API, plataforma o permisos.
  • Escalar con evidencia mínima: producto, periodo, componente, error y captura.
Runbook 02 · Dashboard Grafana no muestra información
  • Validar rango de tiempo seleccionado y variables del dashboard.
  • Ejecutar la consulta KQL directamente en Azure Log Analytics.
  • Revisar permisos del usuario, datasource y workspace.
  • Validar si la función KQL existe en el workspace correcto.
  • Confirmar si hubo cambios recientes en queries, funciones o variables.
Runbook 03 · Error de permisos en Azure
  • Identificar recurso, grupo de recursos, suscripción y acción bloqueada.
  • Validar roles asignados al usuario o grupo.
  • Comparar permisos actuales versus permisos requeridos por el flujo operativo.
  • Escalar a plataforma con evidencia del error y justificación operacional.

Gobierno, calidad y mejora continua

Define controles mínimos para que el soporte sea trazable, auditable, reutilizable y sostenible en el tiempo.

Control operacional

Controles mínimos

  • Todo producto crítico debe tener dashboard, runbook y matriz de escalamiento.
  • Todo incidente relevante debe cerrar con causa, evidencia y acción preventiva.
  • Las consultas KQL críticas deben estar documentadas y versionadas.
  • Los accesos deben revisarse periódicamente según necesidad operacional.

Indicadores de gestión

  • Tiempo medio de detección.
  • Tiempo medio de diagnóstico.
  • Tiempo medio de resolución.
  • Incidentes repetidos por causa raíz.
  • Runbooks actualizados versus pendientes.

Capacitación y onboarding

Ruta sugerida para que nuevos integrantes entiendan el contexto, herramientas, productos y criterios de diagnóstico.

Formación soporte

Módulo 1 · Contexto

Productos soportados, arquitectura general, roles del equipo, responsabilidades y límites de soporte.

Módulo 2 · Herramientas

Azure Portal, Log Analytics, Grafana, DevOps, Teams, ITSM y repositorios documentales.

Módulo 3 · Diagnóstico

Cómo leer logs, validar datos, revisar ejecuciones, distinguir causa técnica versus funcional y escalar correctamente.

Recomendación: cada capacitación debe cerrar con un caso práctico, revisión de dashboard, lectura de logs y simulación de escalamiento.
